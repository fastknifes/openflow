# 工程哲学

本文档阐述 OpenFlow 设计决策背后的 rationale——为什么它这样工作，而不是其他方式。

## 苏格拉底式工程方法

OpenFlow 更接近一种苏格拉底式工程方法：

1. **不急于生成代码** —— 先把问题定义清楚
2. **先画变更边界** —— 哪些能改，哪些不能动
3. **显式标记不可变约束** —— 哪些事实、语义、约束不允许静默改变
4. **先定义什么算完成** —— 在开始之前就说清楚证据标准

大多数工具假设用户已经知道应该构建什么。OpenFlow 显式处理一个更现实的问题：

> 用户描述了一个症状。这个症状意味着 Bug、数据异常、配置漂移、环境故障，还是仅仅需求不清晰？

在边界清晰之前就写代码，往往只是以更快的方式向系统注入不确定性。

## 行为文档约束的 Rationale

### 问题：文档与代码的脱节

在传统开发中，文档和代码是两个独立的体系：

- 文档写完就被遗忘
- 代码持续变化，文档不再反映现实
- AI Agent 没有动力去遵守文档中的约束

### 解决方案：文档即行为契约

OpenFlow 的行为文档约束不是"建议"，而是**契约**：

- `docs/current/` 中的事实 → AI 执行时必须遵守的约束
- `docs/changes/` 中的设计 → 变更边界，AI 不能越界
- `docs/decisions/` 中的决策 → 技术选择的约束条件
- `docs/archive/` 中的记录 → 不可变的历史权威

**为什么这样设计？**

因为 AI Agent 需要明确的边界才能可靠工作。模糊的指令导致不可预测的行为。如果文档只是"参考"，AI 会自行决定何时遵守何时不遵守——这不是治理。

## 证据先于声明的 Rationale

### 问题："AI 说做完了"

AI Agent 的"完成"声明不可靠：

- "I think it is fixed" 不等于真的修好了
- "the agent said it ran" 不等于验证通过
- 口头确认不等于可追溯的证据

### 解决方案：证据门控

OpenFlow 的质量门要求：

- lint、typecheck、test 全部通过
- 变更复杂度评估（决定是否需要对抗性硬化）
- 明确的就绪分类（Ready / ReadyWithDocUpdates / NotReady / NeedsDecision）

**为什么不用更简单的验证？**

因为不同复杂度的变更需要不同强度的验证。简单的配置修改不需要对抗性测试，但涉及多模块的架构变更需要。一刀切的验证标准要么太轻（高风险变更漏检），要么太重（低风险变更被拖慢）。

## 归档即权威边界的 Rationale

### 问题："Done"的歧义

在 AI 辅助开发中，"做完了"有多种含义：

- AI 说做完了（口头声明）
- 代码能跑了（功能性验证）
- 通过了测试（质量验证）
- 变更被追溯和冻结（治理完成）

### 解决方案：归档作为正式关闭阶段

`/openflow-archive` 不是一个简单的文件移动操作：

1. `verify` 产出证据和就绪判定
2. `archive` 冻结历史、更新 current 事实、生成实施映射
3. "完成"是一个明确的工程状态，不是对话中的声明

**为什么归档这么重要？**

因为没有归档，就没有权威的历史记录。代码改了就改了，下次遇到类似问题时，没有可靠的上下文来理解"为什么当时这样改"。归档让每一次变更都有据可查。

## 检查点式漂移检测的 Rationale

### 问题：如何防止代码与设计漂移？

持续监控成本太高（每次文件保存都运行 LLM 检查），而完全不监控则会导致漂移累积到不可挽回。

### 解决方案：检查点式检测

- 设计阶段生成合约标记
- 实施计划中嵌入检查点任务
- 里程碑完成后运行异步语义检查
- 检查结果仅供参考，不替代 quality-gate 的最终判定

**为什么不持续监控？**

因为在编码过程中，代码频繁变化，中间状态的漂移检查会产生大量误报。只在里程碑节点检查，既减少了噪音，又不遗漏关键漂移。
