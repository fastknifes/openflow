# 与竞品对比

本文档提供 OpenFlow 与类似工具（OpenSpec、GSD、Superpowers 等）的对比，帮助你判断 OpenFlow 是否适合你的场景。

::: warning 诚实的声明
OpenFlow 不是最快的 AI 工作流，也不是最轻量的。它为特定类别的问题而设计——如果你需要的是"5 分钟上手"的体验，其他工具可能更适合。
:::

## 定位差异

这些工具从远处看相似（都围绕 AI 开发工作流），但实际解决的问题类别不同：

| 维度 | OpenSpec / GSD 更强的地方 | OpenFlow 更强的地方 |
|------|--------------------------|---------------------|
| 启动速度 | 更快、更轻量，5 分钟即可采用 | 设置成本更高，因为治理模型本身有成本 |
| 最佳场景 | 目标已明确的特性开发 | 模糊问题调查 + 受治理的实施 |
| 完成定义 | 计划 → 执行 → 实现 | 证据 → 就绪判定 → 归档权威 |
| 棕地治理 | 有帮助，但不是核心卖点 | 采用的核心原因之一 |
| 团队持续性 | 更聚焦于单次变更流程 | 通过 current/decisions/archive 构建持久共享记忆 |

**更准确的表述不是"OpenFlow 替代 OpenSpec / GSD"，而是：**

- OpenSpec / GSD 帮助组织**已理解目标**的实施
- OpenFlow 帮助组织**问题边界、验证证据、归档追溯**的受治理变更

## 适用场景判断

### OpenFlow 更适合的场景

- 棕地项目，有生产流量、历史行为和遗留约束
- 团队需要一个统一的循环：需求澄清 → 问题调查 → 验证证据 → 归档追溯
- 架构师、技术负责人和平台团队，关心治理、可审查性和交接质量
- 真实工程场景中，症状可能是 Bug、数据异常、配置漂移、环境故障或语义歧义

### 其他工具可能更适合的场景

- 个人原型项目，首次输出速度最重要
- 团队只想要一个轻量级的规格层
- 团队不打算维护 `docs/current`、`docs/changes`、`docs/archive`
- "代码能跑"就足够，不需要正式的证据/归档权威

## 核心差异项

| 特性 | OpenFlow | 典型竞品 |
|------|----------|----------|
| Issue 调查工作流 | ✅ 内置（分类后才决定下一步） | ❌ 通常默认为 Bugfix |
| 质量门 | 证据级验证 + 就绪分类 | 通常依赖人工检查 |
| 归档追溯 | 自动生成 implementation-mapper.md | 通常无 |
| AI 自我反思 | 内置反思机制，记录可复现的错误模式 | 通常无 |
| 漂移检测 | 基于检查点的合约标记检测 | 通常无或需额外配置 |

## 总结

选择工具的关键不在于"哪个更好"，而在于**你要解决的问题类别**：

- **目标清晰 → 快速实现**：OpenSpec / GSD 可能更轻量
- **边界模糊 → 澄清 → 受治理变更 → 可追溯归档**：OpenFlow 是更合适的工具
