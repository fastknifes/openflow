# Issue 上下文处理

历史上 OpenFlow 提供 `/openflow-issue` 来处理“不确定性质的问题”：先澄清、调查、分类，再决定是否修复。

当前官网不再把它作为独立主工作流推广，因为完成链路已经统一到：

```text
issue context / fix evidence → openflow-quality-gate → /openflow-archive
```

## 什么时候还需要 Issue 上下文？

- 用户只描述了异常现象，不确定是不是 bug；
- 可能是数据、配置、环境或业务语义问题；
- 修复前需要明确“不能改什么”；
- 需要把调查结果作为质量门和归档的上下文证据。

## 核心原则

1. 不把问题报告默认当 bugfix。
2. 调查期只读，不先改代码或数据。
3. 先对齐当前语义，再决定修复。
4. 修复完成后仍回到质量门和归档。

## 与归档的关系

当一次问题处理完成后，归档应该能记录：

- 现象是什么；
- 根因是什么；
- 修复保护了哪些语义；
- 哪些事实需要进入 `docs/current/`；
- 哪些规则只是候选，不能自动生效。

也就是说，Issue 的价值不是“另一套流程”，而是为统一完成链路提供更可靠的上下文。
