# Quality Governance

## Overview

Quality governance is the post-implementation quality gate in OpenFlow. After an AI agent completes code changes or bug fixes, it invokes the `openflow-quality-gate` Skill. This Skill decides whether adversarial hardening is needed, performs evidence-aware verification, and reports readiness.

The quality gate replaced the older manual commands `/openflow-harden` and `/openflow-verify`. Those commands' capabilities are now internal stages coordinated by the quality gate. See ADR-004 for the decision rationale.

---

## Architecture

### Quality Gate as the Central Node

```text
AI completes code or bug fix
  → AI invokes openflow-quality-gate Skill
    → build context bundle (design docs, issue clarification, plan, or git diff)
    → assess change risk → decide whether to run harden
    → perform evidence-aware verify (validate freshness and coverage of existing evidence)
    → output readiness classification
  → archive may proceed after readiness allows
```

The quality gate is registered as a Skill (not only a Command). This is an explicit exception to ADR-003's general rule against auto-triggering governance commands. The quality gate must be AI-callable because it is a post-implementation responsibility, not an interactive design step.

### Superseded Commands

`/openflow-harden` and `/openflow-verify` are no longer the normal manual workflow entrypoints.

- Their underlying implementation (`handleHarden`, `handleVerify`) may remain as internal building blocks.
- The user-facing and AI-facing workflow routes through `openflow-quality-gate`.
- Plans generated by `/openflow-writing-plan` instruct agents to call `openflow-quality-gate` after implementation, not to run `/openflow-harden` or `/openflow-verify` manually.
- Existing `execution-policy.json` fast/balanced/strict state is deprecated historical context.

## Context Detection

The quality gate builds a context bundle from the best available source:

| Context Source | Use Case | Alignment Behavior |
|---|---|---|
| `docs/changes/<feature>/design.md` + `behavior.md` | Feature work | Check design/behavior alignment |
| `issue-clarification.md` | Issue investigation/fix | Check issue context alignment |
| `.sisyphus/plans/<feature>.md` | Implementation guidance | Supporting context, not sole truth source |
| User request + git diff only | Lightweight ad-hoc fix | Limited context alignment |
| No semantic context | Emergency/local change | Technical verification only, limited-confidence warning |

Missing `issue-clarification.md` does not skip verify. It only downgrades the semantic alignment portion.

Missing workflow artifacts (`plan.md`, `design.md`, `behavior.md`, `issue-clarification.md`) are state blockers. The quality gate reports the blocker and suggests the explicit workflow command rather than generating unrequested artifacts.

## Risk Assessment and Harden Decision

The quality gate decides whether to run harden based on change risk:

| Change Risk | Harden Behavior |
|---|---|
| Trivial documentation/formatting/small localized fix | Skip harden |
| Design-only, planning-only, metadata-only, or feature-renaming work | Do not invoke quality gate unless explicitly requested for readiness/archive |
| Simple single-module code fix | Usually skip harden |
| Multi-file logic change, state change, command lifecycle change, public API change | Run harden |
| Security, auth, permission, payment, archive, verify, harden, config, hooks | Run harden |
| Production or data-loss risk | Run harden |

The user is not asked whether to harden. The skill reports the decision and rationale.

## Harden Orchestration

When the quality gate decides harden is needed, it runs the adversarial review loop internally. Harden uses a three-role architecture with independent sessions and system prompts.

### Three Roles

| Role | Responsibility | Authority |
|---|---|---|
| Coordinator | Orchestrate rounds, manage sessions, pass context, judge convergence | Does not review or modify code |
| Reviewer | Review implementation against design docs, output findings | Read-only, cannot modify code |
| Executor | Respond to findings with accept/reject/partial, do minimal fixes | Can modify code within plan scope |

**Coordinator identity**: When OMO is present, the Sisyphus main agent fulfills the Coordinator role directly (no extra sub-agent layer). When OMO is absent, the local agent fulfills it. This avoids unnecessary multi-layer delegation.

**Model selection**: Reviewer uses the highest-quality available model (OMO reviewer sub-model if available). Executor uses a cost-effective execution model (OMO executor sub-model if available, otherwise a cheaper option like deepseek/glm/kimi).

### Complexity Grading

| Level | Criteria | Behavior |
|---|---|---|
| Trivial | Single file, 10 lines or fewer, pure docs/formatting/test/docs | Reject, do not consume review tokens |
| Simple | Single file, 50 lines or fewer, no significant logic branches | Single-round Reviewer + one-time Executor confirmation, no multi-round fix loop |
| Complex | Multi-file with logic branches, state changes, public API changes | Full multi-round adversarial loop |

### Adversarial Loop (Complex Mode)

```text
Round N:
  1. Coordinator creates Reviewer sub-session
     Title: "Harden Round N/{maxRounds} - Reviewer"
     Input: plan summary, current diff, previous disposition report, rebuttal history

  2. Reviewer outputs findings[]
     Format: Level / Description / Evidence / Files

  3. Coordinator filters findings
     - Discard style_or_preference
     - If no findings remain → PASS

  4. Coordinator creates Executor sub-session
     Title: "Harden Round N/{maxRounds} - Executor"
     Input: findings + plan summary + allowed file list

  5. Executor outputs Disposition Report
     Per finding: verdict (accept/reject/partial) + rationale + fix diff (if accept/partial)

  6. Coordinator classifies dispositions
     ├─ All accept/partial → record fix, proceed to Round N+1
     ├─ Has reject → trigger rebuttal sub-flow
     └─ design_ambiguity → stop immediately

Rebuttal sub-flow (max 2 rounds per finding):
  R1: Coordinator sends rejection rationale to Reviewer
      Reviewer: accept rebuttal (withdraw) OR challenge with new evidence
  R2: Coordinator sends Reviewer challenge to Executor
      Executor: final verdict (accept/reject)
      If still reject → Coordinator marks as design_ambiguity, stop that finding
```

### Finding Classification

| Level | Definition | Action |
|---|---|---|
| `blocking_bug` | Will cause functional error, regression, data corruption, or command failure | Auto-fix via Executor |
| `spec_violation` | Implementation deviates from explicit plan/design/decisions constraints | Auto-fix via Executor |
| `regression_risk` | Evidence suggests potential breakage of existing behavior | High-confidence only: auto-fix or add tests. Requires three-part rationale: affected path + trigger condition + expected consequence |
| `test_gap` | Insufficient test coverage, but implementation not proven wrong | Report, no forced implementation change |
| `design_ambiguity` | Design docs do not cover this case | Stop, escalate to user |
| `style_or_preference` | Style, naming, or abstraction preference | Discard |

Only `blocking_bug`, `spec_violation`, and high-confidence `regression_risk` drive Executor fixes. `test_gap` and `design_ambiguity` are reported but not auto-fixed. `style_or_preference` is discarded.

### Convergence Rules

| Condition | Result |
|---|---|
| No findings | `pass` |
| 2 consecutive rounds with only non-blocking findings | `pass_with_risks` |
| `maxRounds` reached | `max_rounds_reached` (blocks as NeedsDecision) |
| `design_ambiguity` (including unresolved rebuttal) | `needs_human` (blocks as NeedsDecision) |
| Trivial | `rejected` (not run) |

### Termination States

| State | Condition | Quality Gate Impact |
|---|---|---|
| `pass` | No blocking/spec_violation/high-confidence regression_risk | Does not block |
| `pass_with_risks` | 2 consecutive rounds with only non-blocking findings | Does not block |
| `max_rounds_reached` | Reached hard round limit | Blocks as NeedsDecision |
| `needs_human` | design_ambiguity / unresolvable / Executor multiple failures | Blocks as NeedsDecision |
| `rejected` | Trivial change | Not run |

### Session and Title Model

```text
Parent session (main workflow / quality-gate session)
  └── handleHarden()
       └── Coordinator (main agent session, no extra sub-agent)
            ├── Round 1 - Reviewer Session
            ├── Round 1 - Executor Session
            ├── Round 1 - Reviewer Rebuttal Session (if rebuttal triggered)
            ├── Round 1 - Executor Rebuttal Session (if rebuttal triggered)
            ├── Round 2 - ...
            └── ...
```

Title formats:
- Reviewer (main round): `Harden Round {round}/{maxRounds} - Reviewer`
- Executor (main round): `Harden Round {round}/{maxRounds} - Executor`
- Reviewer (rebuttal): `Harden Round {round}/{maxRounds} - Reviewer Rebuttal`
- Executor (rebuttal): `Harden Round {round}/{maxRounds} - Executor Rebuttal`

### Harden Report Contract

Harden returns a markdown report to the quality gate with three layers:

**Base metadata (required)**: status, stop_reason, rounds_executed, total_tokens_consumed, coordinator_session_id.

**Per-round details (required)**: reviewer_session_id, findings list (id, level, description, evidence, files), executor_session_id, disposition report (verdict, rationale, fix_summary), rebuttal records if triggered.

**Findings final summary (required)**:

| Category | Impact on Readiness |
|---|---|
| `resolved_findings` | Does not block |
| `rejected_findings` (executor effectively vetoed) | Does not block |
| `unresolved_must_fix` | Blocks as NotReady |
| `unresolved_needs_decision` (design_ambiguity) | Blocks as NeedsDecision |
| `accepted_known_issues` | Downgrades to ReadyWithDocUpdates |

The report format is markdown (backward-compatible with existing `parseFindingsSummary` and `extractHardenStatus` parsers).

### Cost Control

- Token hard limits (`tokenBudgetTotal`, `tokenBudgetPerRound`) are removed. Termination is by `maxRounds` only.
- Token consumption is tracked and reported (pure statistics, no thresholds).
- Old config fields `tokenBudgetTotal` and `tokenBudgetPerRound` are silently ignored if present.

### Review Baseline

Reviewer findings must be anchored exclusively to:

1. Current feature's design/plan/spec (`docs/changes/{feature}/`)
2. Current facts (`docs/current/*`)
3. Global constraints (`docs/decisions/*`)
4. Actual git diff (filtered for formatting and unrelated changes)
5. Relevant test and verification output

Reviewer must not propose new requirements based on personal preference or unanchored best practices. Design-ambiguity findings must be reported as `design_ambiguity`, not implemented directly.

## Evidence-Aware Verify

After harden (or directly, if harden was skipped), the quality gate runs evidence-aware verify.

### Evidence Gate Logic

```text
collect existing evidence
  → check freshness (after last code change)
  → check coverage (against changed files/symbols)
  → reuse fresh sufficient evidence
  → classify missing, stale, or insufficient evidence as readiness blocker or advisory gap
  → output readiness
```

Fresh evidence includes typecheck, test, lint, build, format, and security scan outputs already produced by the implementing agent or CI. The quality gate validates evidence freshness and coverage; it does not directly execute generic unit tests, lint, typecheck, build, format, or framework-specific commands. If evidence is stale, missing, failed, or not scoped to the changed area, the quality gate reports a readiness blocker for required evidence or an advisory gap for optional evidence, rather than running the check itself.

### Integration Evidence and Behavior Coverage

When `behavior.md` exists with critical scenarios, the quality gate checks integration evidence coverage:

- Each critical scenario must have a corresponding integration evidence mapping.
- Evidence can come from automated integration tests, API smoke tests, QA output, or manually reproducible verification records.
- Coverage levels: `exact`, `equivalent`, `partial`, `missing`, `not_applicable`.
- Only `exact` and `equivalent` pass for critical scenarios. `partial` and `missing` block readiness.
- `equivalent` coverage must include a rationale explaining why the semantics match.
- One integration test can cover multiple scenarios, but each scenario must have its own mapping row.

The quality gate does not hardcode test commands or framework choices. It validates coverage, not execution mechanism.

### Compilation Health Probe

For compiled languages (TypeScript, Go, Rust, etc.), a zero-config compilation health probe (e.g., `tsc --noEmit`, `go build ./...`, `cargo check`) is an advisory-only exception path, not part of the core evidence gate. Probe results never replace required persistent evidence and do not directly block readiness: success may support freshness signals, while failure or missing tooling is reported as an advisory gap. The quality gate's core constraint remains evidence coverage validation, not generic command execution.

## Readiness Classification

The quality gate outputs readiness with evidence and confidence:

| Status | Meaning |
|---|---|
| `Ready` | Evidence is fresh and sufficient; context alignment passed or not required |
| `ReadyWithDocUpdates` | Evidence passed but non-blocking risks or advisory gaps exist |
| `ReadyWithLimitedContext` | Technical evidence passed, but no design/issue context was available for full semantic alignment |
| `NotReady` | Blocking evidence failed, or harden found unresolved issues, or critical behavior lacks integration evidence |
| `NeedsDecision` | Design ambiguity, unclear issue semantics, or unresolvable harden finding |

Readiness outcomes for integration evidence:

| Evidence State | Outcome |
|---|---|
| All critical behaviors are exact/equivalent + fresh + verified | Ready |
| Any critical behavior is partial/missing/failed/stale | NotReady |
| Critical evidence is ambiguous or freshness unknown | NeedsDecision or NotReady |
| Only optional/boundary evidence is missing | Ready or ReadyWithDocUpdates with advisory gap |
| not_applicable without reason | NeedsDecision |

## Stage Applicability

The quality gate applies to specific workflow stages:

| Stage | Quality Gate Behavior |
|---|---|
| After feature implementation | Full quality gate (risk assessment + harden if needed + evidence-aware verify) |
| After bug fix | Full quality gate, with degraded context if no design docs exist |
| After issue fix (`--resolve`) | Full quality gate, using issue-clarification.md if available |
| Design-only changes | Do not invoke unless explicitly requested |
| Planning-only changes | Do not invoke unless explicitly requested |
| Metadata-only or feature-renaming | Do not invoke unless explicitly requested |
| Documentation-only changes | Usually skip; invoke only if explicitly requested for readiness/archive |

---

## Dirty Worktree and Plan Evidence Checks

Before running the quality gate, the following preconditions are checked:

1. **Git worktree must be clean or have expected changes only**: If the worktree has unexpected uncommitted changes, the quality gate reports a warning. Harden refuses to run if the git state changes during execution (immediate stop).

2. **Plan evidence must exist for feature work**: For feature implementations, the quality gate expects `docs/changes/{feature}/` to contain plan, design, or behavior documents. Missing artifacts are reported as blockers, not silently skipped or auto-generated.

3. **Evidence freshness must be verifiable**: Evidence without freshness metadata (timestamp, git HEAD) is treated as `unknown`, which does not satisfy critical scenario requirements.

## Configuration

| Config Key | Type | Default | Description |
|---|---|---|---|
| `harden.enabled` | boolean | true | Enable or disable harden stage |
| `harden.maxRounds` | number | 5 | Hard round limit for adversarial loop (deep mode default: 8) |
| `harden.reviewerModel` | string | (system default) | Model for Reviewer role |
| `harden.executorModel` | string | (system default) | Model for Executor role |
| `harden.maxArgumentRoundsPerFinding` | number | 2 | Max rebuttal rounds per finding |

Deprecated config keys (silently ignored): `harden.tokenBudgetTotal`, `harden.tokenBudgetPerRound`.

## Edge Cases

| Case | Behavior |
|---|---|
| Design docs missing | Reject if plan does not exist; review only explicit constraints if plan is incomplete; mark uncovered issues as `design_ambiguity` |
| Executor scope drift | Executor modifies file outside plan scope → mark as scope_violation → reject that round's fix |
| Model unavailable | Degrade to fallback model and report; all unavailable → fail |
| Concurrent modifications | Git state changes during harden → stop immediately |
| Sensitive files | Refuse to review/modify files containing secrets or certificates → `security_sensitive_skip` |

## Risks and Mitigations

| Risk | Mitigation |
|---|---|
| AI forgets to invoke the skill | Skill registered in available skills; writing-plan instructions require it after implementation |
| False positives waste tokens | Findings must include evidence; Coordinator filters evidence-free findings |
| Fix oscillation | Convergence rules prevent indefinite loops; same finding appearing 2+ times → unresolvable |
| Executor scope drift | Only plan-scope files allowed; Coordinator validates fix diff |
| Heterogeneous models share blind spots | Evidence anchoring + design doc baselining, not relying solely on model diversity |
| Hardening runs too often | Explicit risk rules and mandatory rationale when harden runs |
| Hardening skipped on complex changes | Mandatory harden triggers for risk-sensitive paths |
| Missing design docs blocks small fixes | Verify degrades to limited context instead of failing on missing docs |
| Duplicate checks waste time | Evidence-aware verify reuses fresh evidence |
| Rebuttal bloat | `maxArgumentRoundsPerFinding` hard limit + Coordinator forced ruling |
