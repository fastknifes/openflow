# OpenFlow 命令仅手动触发注册设计

**日期**: 2026-05-08  
**Feature**: command-only-registration  
**状态**: Design

---

## 1. 背景与问题

OpenFlow 当前把 `/openflow-brainstorm`、`/openflow-verify`、`/openflow-archive`、`/openflow-init`、`/openflow-migrate-docs` 等治理入口注册为 OpenCode skill。这个做法带来了一个结构性副作用：

> OpenFlow 命令既出现在 OpenCode 的 `/` 命令提示中，也出现在 Agent 的 `<available_skills>` 中。

这会让 Agent 在用户没有显式执行命令时，因为 “brainstorm”、“verify”、“archive” 等语义自动加载或调用 OpenFlow skill。实际体验表现为：

1. 用户只是讨论需求，Agent 自动进入 brainstorm；
2. 用户只是要求验证或总结，Agent 自动调用 verify；
3. 系统提示或 background reminder 中出现关键词时，也可能诱发不必要的 OpenFlow 命令执行；
4. 用户难以区分“OpenFlow 建议我运行命令”和“Agent 已经替我运行命令”。

根因不是 OpenFlow 命令本身，而是 **command、skill、tool 三种暴露方式被混在一起**。

---

## 2. 外部系统事实

本设计基于对 `F:\ai-code\opencode` 与 `F:\ai-code\oh-my-openagent` 的源码阅读，确认以下事实。

### 2.1 OpenCode 支持 command 目录

OpenCode 的 `packages/opencode/src/config/command.ts` 会扫描：

```text
{command,commands}/**/*.md
```

因此以下位置的 Markdown 文件会被加载为 command：

- 全局：`~/.config/opencode/commands/*.md`
- 项目：`.opencode/commands/*.md`

这些 command 会进入 OpenCode 统一命令列表，并在 UI 的 `/` 弹出菜单中以 `source: "command"` 展示。

### 2.2 OpenCode 会把 skill 自动提升为 slash command

OpenCode 的 `packages/opencode/src/command/index.ts` 会把 `skill.all()` 的结果合并进命令列表：

```typescript
for (const item of yield* skill.all()) {
  if (commands[item.name]) continue
  commands[item.name] = {
    name: item.name,
    description: item.description,
    source: "skill",
    template: item.content,
    hints: [],
  }
}
```

这意味着：**注册为 skill 会自然获得 slash command 入口，但也会让 Agent 看到该 skill。**

### 2.3 OpenCode plugin tool 仍是程序执行入口

OpenFlow 在 `src/index.ts` 中通过 `tool()` API 注册真实可执行工具，例如：

- `openflow-brainstorm`
- `openflow-verify`
- `openflow-archive`
- `openflow-init`
- `openflow-status`
- `openflow-config`
- `openflow-migrate-docs`

这些 tool 是业务逻辑执行入口，不能用 command Markdown 直接替代。

### 2.4 OMO 也支持 commands 与 skills 分流

OMO 会发现 `.claude/commands/`、`.opencode/commands/`、`.opencode/skills/` 等来源，并把 commands 与 skills 合并到 OpenCode 的命令体系中。其行为进一步证明：

- command 适合用户显式触发；
- skill 适合 Agent 可主动加载的能力；
- 两者最终都可能显示在 slash command UI 中，但语义边界不同。

---

## 3. 边界约束

### 3.1 必须保持的能力

1. 用户输入 `/openflow-brainstorm <feature>` 时，仍能启动 brainstorm 工作流；
2. 用户输入 `/openflow-verify <feature>` 时，仍能执行验证流程；
3. `/` 弹出菜单中仍能看到 OpenFlow 命令提示；
4. OpenFlow 的 tool handlers 继续作为真实执行入口；
5. `auto_trigger` 只保留“建议用户运行命令”的提示能力，不自动执行；
6. 命令命名继续遵守 ADR-002：使用 `/openflow-brainstorm`，禁止 `/openflow/brainstorm`。

### 3.2 必须移除的能力

1. OpenFlow 命令不再写入 `~/.config/opencode/skills/openflow-*`；
2. OpenFlow 命令不再出现在 `<available_skills>`；
3. Agent 不应因为关键词、背景提醒、验证语境自动加载 OpenFlow skill；
4. `src/command-registration.ts` 不应继续只是 legacy cleanup，却让人误以为 command 已注册。

### 3.3 不属于本次范围

1. 不修改 OpenCode 或 OMO 源码；
2. 不改变 OpenFlow 各命令的业务逻辑；
3. 不删除 `tool()` 注册，因为 tool 是程序执行入口；
4. 不引入全局配置开关矩阵；
5. 不影响非 OpenFlow 的用户 skills；
6. 不要求所有 OpenCode 插件采用同样策略。

---

## 4. 模糊点与决策

### 4.1 “命令提示”来自 command 还是 skill？

**模糊点**：当前 skill 也会显示为 slash command，所以容易误以为必须注册 skill 才能有 `/` 提示。

**决策**：OpenFlow 的 `/` 提示应来自 `commands/*.md`，不是 `skills/*/SKILL.md`。

理由：command source 能在 UI 中显示，但不会进入 `<available_skills>`。

### 4.2 command Markdown 是否能替代 tool？

**模糊点**：OpenCode command Markdown 是模板，不是代码执行器；OpenFlow 命令需要执行 TypeScript handler。

**决策**：不能替代。保留 `tool()` 注册作为执行入口，command Markdown 只负责 slash command 可见性和用户显式触发语义。

### 4.3 用户输入 `/openflow-*` 后由谁执行？

**模糊点**：仅有 command Markdown 时，OpenCode 会把命令变成用户消息，仍需要某个机制把它转成 tool 调用。

**决策**：OpenFlow 应在 `chat.message` hook 中识别 `/openflow-*` 用户消息，并路由到现有 handler；command Markdown 用于 UI 提示与参数说明。

这样避免把执行责任交给 Agent 自行判断，也避免“AI 看到模板后再决定是否调用 tool”的不确定性。

### 4.4 tool 仍然能被 AI 调用，是否违反目标？

**模糊点**：即使移除 skill，plugin tool 仍在工具列表里。

**决策**：本次目标不是让 tool 对 Agent 完全不可见，而是移除 skill 级别的自动发现与语义触发。Tool 保留是 OpenFlow 程序执行的必要条件。

如果后续仍发现 Agent 在未被用户明确要求时直接调用 tool，再单独设计 tool-level permission 或 prompt hardening；本次不扩大范围。

### 4.5 command 文件写到全局还是项目目录？

**模糊点**：OpenCode 同时支持全局 `~/.config/opencode/commands/` 和项目 `.opencode/commands/`。

**决策**：优先采用全局命令目录，与当前 skill 注册目标保持一致：

```text
~/.config/opencode/commands/openflow-brainstorm.md
```

同时清理旧的项目局部遗留文件，避免重复命令或命名冲突。

## 5. 目标架构

### 5.1 暴露层职责

```text
┌───────────────────────────────────────────────────────┐
│ OpenCode Slash UI                                      │
│ source: command                                        │
│ /openflow-brainstorm, /openflow-verify, ...            │
└──────────────────────────┬────────────────────────────┘
                           │ 用户手动输入
┌──────────────────────────▼────────────────────────────┐
│ OpenFlow chat.message hook                             │
│ 解析 /openflow-* 与参数                                 │
└──────────────────────────┬────────────────────────────┘
                           │ 路由
┌──────────────────────────▼────────────────────────────┐
│ Existing OpenFlow command handlers                     │
│ handleBrainstorm / handleVerify / handleArchive / ...  │
└──────────────────────────┬────────────────────────────┘
                           │ 执行
┌──────────────────────────▼────────────────────────────┐
│ OpenFlow tool handlers / workflow state                │
└───────────────────────────────────────────────────────┘
```

### 5.2 注册策略

| 暴露物 | 保留 | 用途 |
|--------|------|------|
| `tool()` definitions | 是 | 程序执行入口 |
| `commands/*.md` | 是 | `/` UI 提示与用户手动入口 |
| `skills/openflow-*/SKILL.md` | 否 | 避免 AI 自动加载 |
| `auto_trigger` 提示 | 是 | 仅建议，不执行 |

## 6. 实现设计

### 6.1 `src/command-registration.ts`

将该文件从 legacy cleanup 改为真实 command 注册模块。

职责：

1. 计算 OpenCode 全局 commands 目录；
2. 写入 OpenFlow command Markdown 文件；
3. 清理旧的 legacy command/skill artifacts；
4. 提供 `registerCommands()` 与 `unregisterCommands()`。

建议导出：

```typescript
export async function registerCommands(): Promise<boolean>
export async function unregisterCommands(): Promise<void>
export async function cleanupLegacyCommandArtifacts(projectDir?: string): Promise<void>
```

命令文件示例：

```markdown
---
description: "OpenFlow brainstorm for design clarification"
---
OpenFlow command: /openflow-brainstorm $ARGUMENTS

This command is handled by the OpenFlow plugin. It must be triggered by the user.
```

### 6.2 `src/index.ts`

修改插件初始化流程：

1. 移除 `registerSkills(openflowCtx)`；
2. 调用 `registerCommands(openflowCtx)`；
3. 保留所有 `tool()` definitions；
4. 保留 `chat.message`、`tool.execute.before`、`tool.execute.after` hooks。

### 6.3 `src/hooks/chat-message.ts`

新增 slash command dispatch：

1. 只处理用户消息以 `/openflow-` 开头的情况；
2. 解析 command name 与 arguments；
3. 路由到现有 handler；
4. 对非 OpenFlow slash command 不做处理；
5. 避免 background/system reminder 触发；
6. 对缺失参数返回帮助信息，而不是猜测 feature。

### 6.4 `src/skills/registration.ts`

本次不需要删除该模块，但 OpenFlow plugin 默认不再调用它。

保留的理由：

- 便于迁移回滚；
- 便于测试 legacy cleanup；
- 避免一次性大拆导致风险扩大。

但必须保证：默认初始化路径不会写入任何 OpenFlow SKILL.md。

## 7. 用户体验

### 7.1 修改前

```text
用户: 我们先验证一下
Agent: [自动加载 openflow-verify skill]
Agent: 开始 OpenFlow verify...
```

### 7.2 修改后

```text
用户: 我们先验证一下
Agent: 建议你手动运行 /openflow-verify <feature>。

用户: /openflow-verify checkout-flow
OpenFlow: [执行 verify handler]
```

### 7.3 `/` 弹出菜单

用户输入 `/openflow` 时应看到：

- `/openflow-brainstorm`
- `/openflow-verify`
- `/openflow-archive`
- `/openflow-init`
- `/openflow-status`
- `/openflow-config`
- `/openflow-migrate-docs`

这些条目的 source 应为 `command`，不是 `skill`。

## 8. 兼容性与迁移

### 8.1 清理旧 skill

初始化时应删除已写入的全局 OpenFlow skill 目录：

```text
~/.config/opencode/skills/openflow-brainstorm/
~/.config/opencode/skills/openflow-verify/
~/.config/opencode/skills/openflow-archive/
~/.config/opencode/skills/openflow-init/
~/.config/opencode/skills/openflow-migrate-docs/
```

否则旧 skill 仍会被 OpenCode 发现，导致 `<available_skills>` 中继续出现 OpenFlow。

### 8.2 清理旧 command 命名

需要清理历史上可能存在的短名或旧路径：

```text
brainstorm.md
verify.md
archive.md
openflow/brainstorm.md
```

目标是只保留中划线命名：

```text
openflow-brainstorm.md
openflow-verify.md
openflow-archive.md
```

### 8.3 对现有用户的影响

用户仍通过同样的 slash command 名称触发命令。变化仅在于：

- Agent 不再通过 skill 自动加载；
- `/` UI 中命令 badge 从 `skill` 语义变为原生 command 语义；
- 命令执行更依赖显式用户输入。

## 9. 测试策略

### 9.1 Command registration tests

验证：

1. `registerCommands()` 写入 `~/.config/opencode/commands/openflow-*.md`；
2. command 文件包含 description；
3. 文件名使用 ADR-002 中划线格式；
4. `unregisterCommands()` 能删除这些文件；
5. legacy `brainstorm.md` 被清理。

### 9.2 Skill removal tests

验证：

1. plugin init 不调用 `registerSkills()`；
2. 初始化后不生成 `~/.config/opencode/skills/openflow-*`；
3. 旧的 OpenFlow skill 目录会被清理；
4. 非 OpenFlow skill 不受影响。

### 9.3 Slash dispatch tests

验证：

1. `/openflow-brainstorm my-feature` 路由到 `handleBrainstorm(ctx, "my-feature", ...)`；
2. `/openflow-verify my-feature` 路由到 `handleVerify(ctx, "my-feature")`；
3. 非 OpenFlow 命令不被拦截；
4. 普通消息不被拦截；
5. system reminder / background message 不触发命令执行；
6. 参数缺失时返回帮助，不自动猜测。

### 9.4 Regression tests

验证：

1. `auto_trigger` 仍只输出建议；
2. brainstorm active session 不因普通关键词自动推进；
3. verify/archive 不因完成语境自动调用；
4. 现有命令 handler 测试保持通过。

---

## 10. 验收标准

1. `available_skills` 中不再出现任何 `openflow-*`；
2. OpenCode `/` 命令提示中仍能看到 OpenFlow 命令；
3. OpenFlow 命令提示的 source 为 `command`，不是 `skill`；
4. 用户显式输入 `/openflow-*` 才会执行对应工作流；
5. Agent 不会因为 brainstorm/verify/archive 关键词自动执行 OpenFlow 命令；
6. 旧的 OpenFlow skill 文件被清理；
7. 现有 OpenFlow tool handlers 仍可正常执行；
8. `bun test` 与 typecheck 通过。

---

## 11. 结论

本设计将 OpenFlow 的用户入口从 “skill 自动提升得到的 slash command” 调整为 “原生 command source 的 slash command”。

核心原则是：

> **OpenFlow commands belong to the user control plane, not the agent auto-skill plane.**

实现上保留 `tool()` 作为程序执行入口，新增真实 command 文件注册作为 UI 入口，并移除默认 skill 注册。这样可以同时满足两个目标：

1. 用户仍能在 OpenCode `/` 菜单中发现并手动触发 OpenFlow 命令；
2. Agent 不再把 OpenFlow 命令当作可自动调用的 skill。
