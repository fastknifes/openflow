# 改进 implementation-mapper 生成质量的提案

- 日期：2026-05-08
- Feature：implementation-mapper-traceability
- 状态：Draft

## 1. 问题陈述

当前自动生成的 `implementation-mapper.md` 有归档留痕价值，但追溯价值偏弱，主要问题如下：

1. 混入临时文件、外部工作区文件、测试运行缓存文件，导致文档噪音过高。
2. “实现映射”只有文件级汇总，没有稳定的符号级映射，难以回答“为什么改这段代码”。
3. 验证证据不够具体，经常只留下“pending”或变更计数，而不是可复核的命令与结果。
4. 文档名叫 implementation mapper，但实际更像变更快照，和 README 中承诺的“requirements → files/functions/symbols”存在落差。

## 2. 目标

本提案希望把 implementation mapper 从“归档附带清单”提升为“可复核的追溯文档”，具体目标是：

- 排除无意义的临时文件与运行噪音。
- 以“需求 / 设计决策 / 实现符号 / 验证证据”为主线生成映射。
- 让后来者能快速理解：改了什么、为什么改、由什么验证。
- 让文档名与文档内容一致，避免名不副实。

## 3. 非目标

以下内容不属于本次提案范围：

- 不尝试构建完整静态分析平台。
- 不要求一次性实现完美的 AST 级需求追踪。
- 不把所有 git diff 行级细节都塞进归档文档。
- 不记录每一次临时实验、每一个缓存产物或每一个 E2E 沙箱文件。

## 4. 核心决策

### 4.1 只收录“仓库内、可复核、与特性有关”的文件

implementation mapper 只应收录以下文件：

- 仓库根目录内的受版本控制文件。
- 与当前 feature 直接相关的源码、测试、文档、配置文件。
- 归档阶段明确声明为证据的生成物。

以下内容必须排除：

- 系统临时目录文件，如 `AppData/Local/Temp/...`
- 一次性测试工作区、沙箱目录、缓存目录
- 与当前 feature 无关、仅因本地环境变化而被触达的文件
- 无法稳定复现来源的外部路径

### 4.2 映射主键从“changed files”改为“需求项 / 设计决策项”

实现映射表不再以一行 `changed files` 汇总所有内容，而应至少按以下维度拆分：

- 需求项
- 设计约束项
- 关键实现决策

每一行都要回答四个问题：

1. 对应哪条需求或设计约束？
2. 落到了哪些文件和符号？
3. 为什么这些代码与该项相关？
4. 用什么证据验证过？

### 4.3 符号级映射优先，文件级映射兜底

优先输出：

- 函数
- 类
- 方法
- 命令入口
- 测试用例名

当无法稳定提取符号时，才退化为文件级映射，但要明确标记为“file-level fallback”，不能用 `-` 直接空置。

### 4.4 验证证据必须具体可复核

验证证据字段必须优先记录：

- 执行命令
- 对应测试文件 / 用例
- 结果状态（passed / failed / skipped）
- 必要时附上证据文件路径

不应再使用如下弱证据：

- `pending`
- 单纯的变更计数
- “已记录”但没有对应验证动作

### 4.5 不把 Git 已有信息伪装成 implementation mapper

Git 已经天然提供了以下能力：

- 变更了哪些文件
- 变更发生在什么提交里
- 文件级 diff 是什么

因此，implementation mapper 不应再把“修改文件清单”当作主体内容重复一遍。

只有在以下场景，才允许留下**最小化的例外引用**，而不是文件清单：

- 某些证据文件不在常规源码目录中，但被明确纳入验收证据
- 某条全局决策或外部证据若不显式指出，会影响读者理解

也就是说，文档应该优先回答“为什么这些实现存在、由什么验证”，而不是重复回答“改了哪些文件”，更不应该为了完整性再输出一份变更清单。

### 4.6 不重复罗列同一 feature 工作区内天然相邻的文档

对于同一 feature 工作区内的文档，例如：

- `proposal.md`
- `design.md`
- `requirements.md`

它们本来就位于同一上下文目录中，implementation mapper 没必要再专门增加一个“关联文档”章节去重复罗列。

真正值得显式关联的，是那些**不在当前 feature 邻近上下文中**、但会影响理解或验收判断的内容，例如：

- `docs/decisions/` 下的 ADR 或全局治理决策
- `docs/current/` 中被当前实现依赖的系统级事实
- 独立保存的 verify evidence
- 少数被显式保留的外部证据文件

因此，implementation mapper 应只保留“全局依赖与例外证据”，而不重复列出同目录下天然可见的 feature 文档。

### 4.7 空章节直接省略，不为模板完整性牺牲可读性

如果某一类信息不存在，就不应该为了模板整齐而输出空章节。

尤其是：

- 没有跨 feature / 全局依赖时，不输出“全局依赖与例外证据”
- 没有需要说明的例外证据时，不输出额外说明块
- 没有已知偏差时，只保留简短结论，不制造空表格

原则是：最终文档服务阅读，不服务模板对称。

### 4.8 映射结果必须稳定排序，避免同样输入生成不同阅读顺序

implementation mapper 应有固定排序规则，避免每次生成都换顺序，影响 diff 可读性。

建议排序优先级：

1. 按 requirements / design 中的原始出现顺序
2. 同一需求项下，先产品代码，再测试，再文档
3. 同类项内部按文件路径与符号名稳定排序

这样可以减少无意义 diff，也方便后续审阅和归档比对。

## 5. 建议的文档结构

建议新的 `implementation-mapper.md` 结构如下：

```md
# {feature} - Implementation Mapper

## 1. 概述
- 本次变更解决的问题
- 归档时间
- 追溯范围

## 2. 全局依赖与例外证据（仅在存在时输出）
- 相关 ADR / 全局决策
- 被当前实现依赖的 current 文档
- 独立 verify evidence
- 例外保留的外部证据

## 3. 需求到实现映射
| 追溯来源 | 需求/决策 | 代码文件 | 关键符号 | 关联说明 | 验证证据 |
|----------|-----------|----------|----------|----------|----------|

## 4. 验证与结论
- 执行过的命令
- 结果
- 尚存偏差
```

## 6. 生成规则提案

### 6.1 文件分类

生成器应先把变更文件分成：

- product code
- tests
- docs
- config
- generated evidence
- excluded noise

### 6.2 排除规则

默认排除以下路径模式：

- `**/AppData/Local/Temp/**`
- `**/tmp/**`
- `**/.tmp/**`
- `**/.cache/**`
- 仓库根目录之外的绝对路径

如果某个外部路径确实必须保留，必须显式标注原因，例如“人工指定的验收证据目录”。

### 6.3 主映射生成顺序

建议按以下顺序构建主映射：

1. 读取 feature 下的 proposal / design / requirements 作为内部输入
2. 提取需求项与关键决策项
3. 将归档变更与仓库内符号做关联
4. 提取相关测试与验证命令
5. 生成人可读的追溯表
6. 仅在存在跨 feature / 全局依赖时输出“全局依赖与例外证据”
7. 按固定排序规则输出，避免无意义顺序漂移

## 7. 风险与取舍

| 风险 | 说明 | 缓解方式 |
|------|------|----------|
| 符号提取不完整 | 某些文档或配置变更无法稳定映射到符号 | 允许 file-level fallback，但必须显式标记 |
| 排除规则过严 | 可能漏掉少数有价值的生成证据 | 引入“显式保留”白名单 |
| 文档变长 | 信息质量提升会带来篇幅增加 | 默认摘要优先，清单次之 |
| 输出顺序漂移 | 相同输入多次生成后 diff 噪音过高 | 固定按需求顺序、文件类型、文件路径排序 |

## 8. 验收标准

- [ ] implementation mapper 不再包含系统临时目录路径。
- [ ] 主映射表至少按需求项或设计决策项拆分，而不是单行汇总。
- [ ] 对于源码变更，优先输出符号名；无法提取时显式标记为 file-level fallback。
- [ ] 验证证据包含具体命令或测试，而不是仅有 pending / count。
- [ ] 文档不再把 Git 已能提供的文件变更清单作为主体重复输出。
- [ ] 文档不输出“排除项”章节，过滤规则仅存在于生成逻辑中。
- [ ] 文档不再重复罗列同一 feature 工作区中天然可见的 proposal / design / requirements。
- [ ] 没有内容的章节不会为了模板完整性而被输出。
- [ ] 相同输入下，映射行顺序稳定，不产生无意义顺序 diff。
- [ ] README 中对 implementation mapping 的承诺与实际产物一致。

## 9. 结论

这次改动的本质不是“删掉临时文件”这么简单，而是重新定义 implementation mapper 的职责边界：

- 它首先应该服务追溯；
- 其次才服务归档审计；
- 不应把噪音路径、缓存产物和弱证据包装成“实现映射”。

只有这样，implementation mapper 才配得上这个名字。
