# Verify 命令重设计

**日期**: 2026-04-21
**Feature**: verify-redesign
**状态**: Design

---

## 1. 问题陈述

当前工作流中，`verify`、`acceptance`、`archive` 三者的边界存在明显重叠与语义漂移风险：

1. `verify` 当前更像“代码层面的检查集合”，主要关注测试、构建、类型检查等证据项；
2. `acceptance` 当前更像“用户层面的验收提示”，承担文档同步、漂移检测、用户验收引导；
3. 但在真实讨论中，诸如 scope 对比、intent vs actual delta、文档对齐等职责，既可被解释为 `verify` 的一部分，也可被解释为 `acceptance` 的一部分；
4. `archive` 又在实践中承担了过多“补录 current、修复状态、冻结历史”的职责，导致 closure 语义混杂。

这暴露出的根本问题不是命令数量，而是：

> **工作流尚未把“事实生产”“准备度判断”“权威发布”“历史冻结”分成清晰的状态迁移。**

如果继续沿用“verify = 技术校验、acceptance = 用户校验”的粗粒度划分，命令边界会持续模糊，scope 对比、文档对齐、漂移处理等职责仍会在多个节点之间来回漂移。

---

## 2. 设计目标

本次设计的目标不是增加更多命令，而是收敛出一个更清晰、更符合第一性原理的 closure 流程。

### 2.1 核心目标

1. **收敛用户入口**
   - 对用户只暴露一个 `Verify` 入口，避免并列校验节点竞争职责。

2. **明确内部阶段**
   - `Verify` 内部明确分为两个阶段：
     - `Evidence`
     - `Readiness`

3. **保持命令语义纯度**
   - `Verify` 不负责 canonicalization，不直接写入 `current`，不承担最终 authority。
   - `Archive` 才是最终 authority，负责 current promotion 与历史冻结。

4. **明确对齐基准**
   - 归档前，`Verify` 主要对齐 `changes/{feature}`。
   - 同时检查是否违反既有 `current/*` 与 `decisions/*` 中的稳定约束。

5. **减少语义漂移**
   - 不再让“是否准备好进入 closure”和“是否已经成为当前真相”在同一命令中混杂。

### 2.2 非目标

1. 不把 `Verify` 变成自动改写所有文档的治理命令。
2. 不让 `Archive` 重复执行低层检查，避免与 `Verify` 重叠。
3. 不在本设计中引入额外的厚重文档类型。
4. 不要求 `Verify` 直接替代完整的业务决策流程。

---

## 3. 核心设计原则

本次设计遵循以下原则：

### 3.1 Facts before authority

系统必须先回答“我们知道了什么”，再回答“这些事实是否足以进入 closure”。

### 3.2 Readiness is not canon

`Verify` 只建立 readiness，不授予 canonical truth。

### 3.3 Archive is the publication authority

`Archive` 负责：

- 判断是否允许正式 closure
- 执行 `current` promotion
- 冻结 archive provenance

### 3.4 One question, one authority

不同问题必须由不同文档层回答：

- 本轮 intent 是什么 → `changes/{feature}`
- 当前系统已接受事实是什么 → `current/*`
- 跨 feature 的长期规则是什么 → `decisions/*`
- 这次变更历史是怎么形成的 → `archive/{feature}`

---

## 4. 概念模型

本次设计将 closure 流程拆成三个不同层级的语义动作：

### 4.1 Evidence

回答：

- 跑了哪些检查
- 观察到了哪些行为
- 当前实现与预期之间存在哪些 delta
- 哪些风险或证据仍缺失

这是**事实生产阶段**。

### 4.2 Readiness

回答：

- 当前变更是否具备进入 closure 的条件
- 是否仍需要补文档
- 是否存在必须先显式决策的问题

这是**准备度评估阶段**。

### 4.3 Canonicalization

回答：

- 这次变更是否被正式接受
- 哪些事实应进入 `current`
- 哪些内容只进入 `archive`

这是**权威发布阶段**，由 `Archive` 承担。

---

## 5. Verify 命令设计

### 5.1 用户形态

`Verify` 对外是**单一入口**，但内部自动执行两阶段流程：

1. `Evidence`
2. `Readiness`

### 5.2 产品定位

- 开发中途可运行，作为健康检查与 readiness 预演；
- 但语义上默认是**完成前主闸门**；
- 它输出的是 closure readiness，不是 canonical truth。

### 5.3 对齐基准

`Verify` 的判断基准分两层：

#### 主对齐源

- `changes/{feature}`

它回答本轮 change 的 intent，包括：

- 本次目标
- 本次不做什么
- 本次约束
- 本次验收预期

#### 稳定约束检查

- `current/*`
- `decisions/*`

它们不与 `changes/{feature}` 争夺本轮语义主权，但用于判断是否存在稳定规则冲突。

### 5.4 Evidence packet

`Evidence` 阶段至少生成以下内容：

1. `checks run / result`
2. `observed behavior summary`
3. `intended vs actual delta`
4. `doc alignment summary`
5. `current/decisions conflict summary`
6. `known risks / missing evidence`

### 5.5 Readiness 输出状态

`Readiness` 阶段输出以下四态：

- `ready`
- `ready_with_doc_updates`
- `not_ready`
- `needs_decision`

#### 状态语义

##### `ready`

当前 evidence 足以表明该 change 已具备进入 Archive closure 的条件。

##### `ready_with_doc_updates`

当前 change 在核心语义上已具备 closure 条件，但仍需要在 closure 前完成明确的文档更新。

##### `not_ready`

当前 evidence 不足或行为/约束不满足，尚不能进入 Archive。

##### `needs_decision`

只用于以下严格情况：

- 涉及 rule 级变化
- 涉及 `current` 级冲突
- 涉及需要显式业务裁决的事项

它**不**用于一般 bug、一般证据缺失、或普通工程不确定性。

### 5.6 用户交互风格

`Verify` 采用**半自动流**：

1. 自动执行 `Evidence`
2. 展示 evidence packet
3. 自动进入 `Readiness`
4. 输出 readiness 状态与下一步建议

对于以下结果：

- `ready_with_doc_updates`
- `needs_decision`

`Verify` 负责：

- 报告问题
- 给出下一步建议

但**不直接吞掉后续文档同步或决策流程**。

### 5.7 Verify 不做什么

`Verify` 不负责：

1. 直接写入 `current`
2. 冻结 archive
3. 赋予 change canonical status
4. 代替用户或 Archive 做最终权威裁决

## 6. Archive 命令设计

### 6.1 定位

`Archive` 是 closure 流程中的**最终 authority**。

它负责：

1. 读取 Verify 结果
2. 作为最终 canonicalization 节点做拍板
3. 执行 `current` promotion
4. 冻结 `archive/{feature}`
5. 生成 implementation mapper / provenance

### 6.2 准入规则

只有以下状态可以进入 Archive：

- `ready`
- `ready_with_doc_updates`

以下状态**不得进入 Archive**：

- `not_ready`
- `needs_decision`

### 6.3 `ready_with_doc_updates` 的处理方式

采用**半自动确认流**：

1. `Archive` 展示拟更新的文档与内容范围
2. 用户确认后再执行文档更新
3. 随后执行 current promotion 与 archive freeze

这保证：

- `Archive` 保持最终 authority 地位
- 文档更新不会变成黑箱操作

### 6.4 Archive 不做什么

`Archive` 不应重复执行低层技术校验。它依赖 `Verify` 的 evidence 与 readiness 结果，而不是重新扮演第二个 `Verify`。

---

## 7. 命令边界

### 7.1 Verify 的边界

`Verify` 负责：

- 事实生产
- readiness 评估
- closure 前状态汇总

`Verify` 不负责：

- canonical acceptance
- `current` promotion
- provenance freeze

### 7.2 Archive 的边界

`Archive` 负责：

- closure authority
- `current` promotion
- archive freeze
- 实现映射与历史追溯

`Archive` 不负责：

- 成为第二个低层检查器
- 重新承担 `Verify` 的 evidence 生产职责

---

## 8. 状态流转模型

推荐状态流转如下：

```text
Intent established in changes/{feature}
  -> Implemented
  -> Verify / Evidence
  -> Verify / Readiness
     -> ready
     -> ready_with_doc_updates
     -> not_ready
     -> needs_decision
  -> Archive
     -> canonicalization
     -> current promotion
     -> archive freeze
```

其中：

- `Verify` 的结果表示“是否准备好进入 closure”；
- `Archive` 的结果表示“是否已经成为 canonical truth 并完成冻结”。

---

## 9. 为什么不采用“Verify 直接拍板”模型

本设计明确不采用“Verify 完成 acceptance / promotion，Archive 只冻结历史”的模型，原因如下：

1. 该模型虽然在状态机上更短，但会让 `Verify` 失去“纯验证”语义；
2. 一旦 `Verify` 拥有 current promotion 的最终决定权，它就不再只是 readiness assessment，而是在做 canonical authority；
3. 对当前工作流目标而言，closure 的庄重性与 authority 语义更适合放在 `Archive`；
4. 因此本设计选择：

> **Verify establishes readiness; Archive makes the change canonical.**

---

## 10. 风险与注意事项

### 10.1 风险：Verify 与 Archive 再次语义重叠

若 `Verify` 使用 `accepted/rejected` 一类命名，或 `Archive` 重新执行低层检查，则会再次引入边界模糊。

**缓解**：

- `Verify` 只使用 readiness 语义
- `Archive` 只承担 authority 与 freeze 语义

### 10.2 风险：用户把 `ready` 误解成“已经完成 canonicalization”

**缓解**：

- 明确文案：`ready` 表示“可进入 Archive”，不是“已完成 Archive”

### 10.3 风险：`needs_decision` 滥用

**缓解**：

- 严格限制其适用范围，仅用于规则级、current 级或业务裁决问题

---

## 11. 预期收益

采用本设计后，工作流将获得以下收益：

1. **单入口更清晰**
   - 用户不再面对多个语义重叠的校验节点。

2. **命令职责更纯**
   - `Verify` 负责 readiness；`Archive` 负责 authority。

3. **减少文档漂移与语义漂移**
   - 不再让“准备度判断”与“正式真相发布”在不同命令间来回混杂。

4. **更符合治理型工作流**
   - closure 由最终 authority 执行，current promotion 不再成为隐式附带动作。

---

## 12. 一句话总结

本次设计将现有模糊的 `verify / acceptance / archive` 关系，收敛为一个更明确的 closure 模型：

> **Verify 建立 readiness，Archive 赋予 canonical status。**

这使得工作流不再围绕“谁在检查”，而是围绕“谁在生产事实、谁在授予权威、谁在冻结历史”来组织。
