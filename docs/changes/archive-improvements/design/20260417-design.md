# Archive 归档命令改进设计

**日期**: 2026-04-17
**Feature**: archive-improvements
**状态**: Design

---

## 1. 问题陈述

当前 `/openflow/archive` 命令存在三个行为缺陷，与 ADR-001 中定义的归档流程设计不一致：

### 1.1 归档后未清理 changes 目录

**现象**: 归档完成后，`docs/changes/{feature}/` 原目录及其文件仍然存在。

**期望**: 归档意味着 feature 工作结束，`docs/changes/` 是"进行中的变更工作区"（ADR-001 §5.1），归档后应清理。

**根因**: `src/commands/archive.ts` 只执行 copy 操作，无删除逻辑。

### 1.2 产生多余空目录

**现象**: `docs/archive/openflow-init/requirements/` 是空目录（0 个文件）。

**期望**: 只有当源目录包含实际文件时，才在 archive 中创建对应子目录。

**根因**: `safeCopyDirectory` 内部先 `mkdir(recursive: true)` 再遍历拷贝，即使遍历结果为空也会留下目录。

### 1.3 未维护 current 目录

**现象**: `docs/current/` 下只有空的 `design/` 目录，归档后的文档没有同步到 current。

**期望**: 归档完成后应将已冻结的文档 promote 到 `docs/current/`，保持 current 作为"当前有效事实"的职责（ADR-001 §5.1）。

**根因**: `auto_promote_current` 默认值为 `false`（types.ts:98），且 promotion 只处理 design/requirements 两个区域。

---

## 2. 设计决策

### 2.1 归档后自动删除 changes/{feature}/

**决策**: 归档完成后直接删除 `docs/changes/{feature}/` 整个目录，不需要用户确认。

**理由**:
- 归档本身就是一个明确的"feature 完成"信号
- 用户已经执行了 `/openflow/archive <feature>` 命令
- ADR-001 明确 changes 是进行中的工作区，归档后不再需要
- 如果用户需要回溯，archive 目录中已有完整快照

**实现要点**:
- 在 `handleArchive` 末尾、返回结果之前执行删除
- 使用 `fs.rm(changeDir, { recursive: true, force: true })`
- 删除前确认 archive 已成功创建（容错）
- 仅删除 `docs/changes/{sanitizedFeature}/`，不影响其他 feature

### 2.2 跳过空目录，不创建多余文件夹

**决策**: 归档时检查源目录是否为空，为空则跳过，不创建对应的目标子目录。

**理由**:
- 空目录没有信息价值，只是噪音
- ADR-001 §5.2 明确不建议在归档中放半成品或空结构
- 用户报告这是一个实际问题

**实现要点**:
- 在 `safeCopyDirectory` 调用前，先检查源目录是否有文件
- 提取一个 `directoryHasFiles(dir)` 工具函数
- 只在有文件时才执行 copy + mkdir
- 对 design/、plans/、requirements/ 三个区域统一应用

### 2.3 默认自动 promote current，懒创建目录

**决策**: 将 `auto_promote_current` 默认值改为 `true`。Promote 时采用懒创建策略——只在需要移动文件时才创建目标目录。

**理由**:
- ADR-001 §11.5 明确 archive 阶段应维护 `current/spec` 和其他 current 子目录
- ADR-001 §5.1 定义 `docs/current/` 为"当前有效事实"
- 懒创建比主动创建更干净，不产生空目录

**实现要点**:
- `types.ts` 中 `defaultConfig.archive.auto_promote_current` 改为 `true`
- `current-promotion.ts` 的 `applyPromotionSuggestions` 中，在拷贝前检查目标目录是否存在，不存在则创建
- `safeCopyDirectory` 已内置 `mkdir(recursive: true)`，无需额外处理

### 2.4 Current 目录扁平存放

**决策**: Promote 到 `docs/current/` 时采用扁平格式，不按 feature 创建子目录。

**格式**: `docs/current/design/20260415-design.md`（非 `docs/current/design/openflow-init/20260415-design.md`）

**理由**:
- 用户明确选择扁平格式
- current 是"当前系统是什么样"的事实集合，不是 feature 维度的索引
- ADR-001 中 current 的目录结构也是 `requirements/`、`design/`、`spec/`、`workflow/` 四个区域

**实现要点**:
- 修改 `current-promotion.ts` 中 `PROMOTION_AREAS` 的 `targetPath` 生成逻辑
- 从 `docs/current/design/{feature}/` 改为 `docs/current/design/`（去掉 feature 子目录层）
- 注意文件名冲突：如果多个 feature 有同名文件，后归档的覆盖先归档的（符合"当前有效"语义）

## 4. 实现步骤

### Step 1: 添加 `directoryHasFiles` 工具函数

在 `src/utils/security.ts` 中添加：

```typescript
export async function directoryHasFiles(dir: string): Promise<boolean> {
  try {
    const entries = await fs.readdir(dir)
    return entries.length > 0
  } catch {
    return false
  }
}
```

并在 `src/utils/index.ts` 中导出。

### Step 2: 修改 `archive.ts` — 空目录跳过

在 design/requirements/plans 的 copy 逻辑前，加入 `directoryHasFiles` 检查：

```typescript
// 替换现有的 designExists 检查
if (designDir && await directoryHasFiles(designDir)) {
  // ... copy logic
}

// 替换现有的 requirementsExists 检查
if (requirementsDir && await directoryHasFiles(requirementsDir)) {
  // ... copy logic
}
```

### Step 3: 修改 `archive.ts` — 删除 changes 目录

在 `generateAndSaveImplementationMapper` 之后、`buildPromotionSuggestions` 之前，添加删除逻辑：

```typescript
// 清理 changes 工作区
const changeDir = createSafePath(ctx.directory, ctx.config.brainstorming.output_dir, sanitizedFeature)
if (await fileExists(changeDir)) {
  await fs.rm(changeDir, { recursive: true, force: true })
  logger.info('Cleaned up changes directory', { path: changeDir })
}
```

### Step 4: 修改 `current-promotion.ts` — 扁平 promote

修改 `buildPromotionSuggestions` 中的 targetPath 生成：

```typescript
// 从 createSafePath(projectDir, area.currentBaseDir, feature)
// 改为 createSafePath(projectDir, area.currentBaseDir)
```

### Step 5: 修改 `types.ts` — 默认值

```typescript
auto_promote_current: true,  // 原来是 false
```

### Step 6: 更新 skill 文档

更新 `src/skills/archive-skill.ts` 和 `SKILL.md`，补充：
- 归档后自动清理 changes 目录
- 默认自动 promote 到 current
- 空目录不创建

## 5. 风险与缓解

| 风险 | 等级 | 缓解措施 |
|------|------|----------|
| 误删 changes 目录中未被归档的文件 | 低 | 删除前确认 archive 已创建成功；用户可通过 git 恢复 |
| 扁平 promote 文件名冲突 | 低 | 后归档覆盖先归档，符合"当前有效事实"语义 |
| `auto_promote_current` 默认值变更影响现有用户 | 低 | 行为变更是用户明确要求的改进；用户仍可通过配置改回 false |
