# openflow-feature 优化决策记录

## Decision: Treat `/openflow-feature <text>` as requirement description, not feature id

Scope: Candidate Global

Alternatives considered:
- Treat `<text>` as a feature id, directory name, slug, or explicit feature parameter.
- Treat `<text>` as natural-language requirement description.

Reason:
Users should not need to remember feature names, directories, slugs, or command parameters. The command is intended to be natural-language driven.

Impact:
Feature identity is generated by AI from the current session context and requirement description. Cross-session continuation by feature name is not supported.

Promotion note:
Update command workflow documentation for `/openflow-feature` and, if needed, ADR-001 command/document governance language.

## Decision: One session can contain only one openflow-feature

Scope: Candidate Global

Alternatives considered:
- Allow multiple features in a single session.
- Allow users to switch feature by invoking `/openflow-feature` again.
- Reject new feature starts once a session already contains a feature.

Reason:
Multiple feature topics in one session pollute context and make Feature Brief, stable decisions, and generated documents unreliable.

Impact:
If a session already contains a feature, another `/openflow-feature ...` invocation is rejected. Users must open a new session for a new feature.

Promotion note:
Update `/openflow-feature` workflow documentation to define the single-feature-per-session invariant.

## Decision: Do not support cross-session feature continuation

Scope: Candidate Global

Alternatives considered:
- Resume old features by passing a name or slug in a new session.
- Scan active or unfinished feature sessions.
- Treat each new session invocation as a new feature.

Reason:
Loading another session’s context risks mixing unrelated discussions and makes user intent ambiguous. The user should only describe current needs in the current session.

Impact:
`/openflow-feature` does not scan other sessions, active features, or unfinished feature sessions. New session invocation starts a new feature.

Promotion note:
Update `/openflow-feature` workflow documentation and current workflow rules.

## Decision: Use dynamic slot convergence instead of fixed questionnaire order

Scope: Local

Alternatives considered:
- Ask `problem`, `target-users`, `scope`, `priority`, `constraints` in a fixed order.
- Infer known slots and ask only the most blocking next question.

Reason:
Fixed-order questioning repeats information users already provided and feels unnatural. Dynamic convergence preserves structure while reducing unnecessary questions.

Impact:
Feature Brief records slot value, confidence, and source. The next question is selected by blocking risk, ambiguity, and confidence.

Promotion note:
Not applicable.

## Decision: Persist Feature Brief in session state only

Scope: Local

Alternatives considered:
- Generate `feature-brief.md`.
- Store data in `design.meta.json` and `behavior.meta.json`.
- Store Feature Brief only in opaque feature session state.
- Store Feature Brief in `docs/changes/{feature}/state.md`.

Reason:
Feature Brief is process memory, not a formal design document. Persisting it in `state.md` keeps the working memory close to the feature workspace without introducing separate meta JSON files or polluting formal documents.

Impact:
Document generation reads structured `state.md` / session state instead of relying on long chat context or meta files. AI updates `state.md` after each answered clarification question.

Promotion note:
Not applicable.

## Decision: Limit AI document edits to the current feature workspace

Scope: Candidate Global

Alternatives considered:
- Allow AI to update any related documentation while editing a feature.
- Restrict AI natural-language document edits to the current feature directory.

Reason:
Natural-language edit authorization should not silently escape into global governance or unrelated feature documents. Global changes belong to archive or explicit governance flows.

Impact:
AI can modify any file under `docs/changes/{feature}/*` for the current feature, but cannot modify files outside that folder during feature document editing.

Promotion note:
Update `/openflow-feature` workflow documentation and archive/global governance boundaries.

## Decision: Support dated and semantic feature references in writing-plan

Scope: Candidate Global

Alternatives considered:
- Require full dated feature directory names.
- Require only semantic feature names.
- Support both full dated directory names and semantic names.

Reason:
OpenFlow feature directories use dated names, but users should not have to remember exact date prefixes. Supporting both forms makes the next command easier while preserving current directory naming.

Impact:
`/openflow-writing-plan 2026-05-25-openflow-feature-optimization` and `/openflow-writing-plan openflow-feature-optimization` should resolve to the same feature when unambiguous.

Promotion note:
Update `/openflow-writing-plan` command documentation and resolver behavior.

## Decision: Make design.md and behavior.md mandatory

Scope: Candidate Global

Promotion status: already reflected in ADR-001 update

Alternatives considered:
- Generate only `design.md` for simple needs.
- Generate `behavior.md` only when behavior is complex.
- Always generate both documents.

Reason:
`design.md` and `behavior.md` form the minimum Cross-Validation loop between design constraints and observable behavior. Stable downstream planning and validation require both.

Impact:
Every feature design must include both documents. A feature design is not complete unless both exist and Cross-Validation passes.

Promotion note:
Update ADR-001 to define `design.md` and `behavior.md` as mandatory `/openflow-feature` outputs.

## Decision: Remove proposal.md from the default openflow-feature flow

Scope: Candidate Global

Promotion status: already reflected in ADR-001 update

Alternatives considered:
- Generate `proposal.md` by default.
- Keep `proposal.md` as a conditional feature document.
- Remove `proposal.md` from the default flow.

Reason:
The current design uses `requirements.md`, `prd.md`, `design.md`, `behavior.md`, and `decisions.md` with clearer responsibilities. `proposal.md` duplicates these documents in the default flow.

Impact:
`/openflow-feature` does not generate, validate, or depend on `proposal.md` by default. Existing global docs mentioning default proposal generation need update.

Promotion note:
Update ADR-001 and current workflow docs to remove `proposal.md` from the default feature document set.

## Decision: Conditional requirements.md and prd.md based on feature type

Scope: Candidate Global

Promotion status: already reflected in ADR-001 update

Alternatives considered:
- Always generate `requirements.md`.
- Always generate `prd.md`.
- Generate one or both based on the current feature’s complexity and primary concern.

Reason:
OpenFlow is a general workflow tool and may be used across projects with unknown complexity. Document generation should follow the feature’s needs rather than the current repository type.

Impact:
Engineering execution features generate `requirements.md`; product-value features generate `prd.md`; complex features may generate both.

Promotion note:
Update ADR-001 document governance matrix.

## Decision: Use local decisions with Candidate Global promotion during archive

Scope: Candidate Global

Promotion status: already reflected in ADR-001 update

Alternatives considered:
- Write global ADR changes during `/openflow-feature`.
- Require user confirmation during `/openflow-archive` for every global promotion.
- Record local decisions during feature design and automatically promote Candidate Global decisions during archive.

Reason:
Feature design should not directly mutate global governance, but global-impact decisions should not be lost. The user is informed during design, so archive can promote without re-asking.

Impact:
`decisions.md` supports `Scope: Candidate Global`. Completion output lists such decisions and explains archive promotion.

Promotion note:
Update archive workflow documentation to promote Candidate Global decisions automatically.

## Decision: Critical Blocking Gaps prevent both completion and draft generation

Scope: Candidate Global

Alternatives considered:
- Allow all Cross-Validation failures to become drafts.
- Block only missing mandatory documents.
- Distinguish Non-blocking, Blocking, and Critical Blocking gaps.

Reason:
Some inconsistencies would lead implementation in opposite or dangerous directions. Allowing them to become drafts would make unsafe or contradictory documents look usable.

Impact:
Critical Blocking Gaps produce `failed` status, block completion, and block draft generation until resolved.

Promotion note:
Update `/openflow-feature` and quality/readiness documentation where document validation semantics are described.

## Decision: AI-managed Cross-Validation Summary in validated documents

Scope: Local

Alternatives considered:
- Store validation state in meta JSON files.
- Store validation state only in session state.
- Write AI-managed Summary blocks into validated documents.

Reason:
Summary blocks make the latest validation result visible in documents without adding separate meta files. AI can rebuild them if missing or damaged.

Impact:
All validated documents include an AI-managed `OPENFLOW:CROSS_VALIDATION_SUMMARY` block.

Promotion note:
Not applicable.

## Decision: Prefer refactoring and rule models over compatibility leftovers

Scope: Local

Alternatives considered:
- Layer new behavior on top of existing command logic while keeping old fallbacks.
- Refactor the feature workflow around explicit rule models, validators, strategies, and state transitions.

Reason:
The optimization changes core semantics: feature identity, document set selection, state persistence, Cross-Validation, and editing authorization. Leaving old behavior as fallback would make the command unpredictable and undermine the new design.

Impact:
Implementation should clean old proposal/meta/fixed-question/cross-session/active-feature logic rather than preserve it. Temporary errors from over-cleanup are preferable to silent historical behavior leaking into the new workflow.

Promotion note:
Not applicable.

## Decision: Enforce deterministic feature workflow order and recovery boundaries

Scope: Local

Alternatives considered:
- Let each command branch decide when to restore state, generate docs, and validate.
- Define a single command execution order with explicit failure boundaries.

Reason:
The optimized workflow has multiple rule systems. Without a fixed order, implementation can accidentally generate documents before detecting existing feature state, or validate before state recovery.

Impact:
`/openflow-feature` must run session detection, `state.md` recovery, Feature Brief update, convergence, document selection, confirmation, write, Cross-Validation, and final output in that order. State write/recovery failure blocks formal generation.

Promotion note:
Not applicable.

## Decision: Require command dispatch and feature state token before document writes

Scope: Candidate Global

Alternatives considered:
- Let AI interpret `/openflow-feature ...` as ordinary chat and write documents directly.
- Require command dispatch and a current feature state token before formal document writes.

Reason:
The feature workflow depends on session isolation, Feature Brief state, confirmation, and Cross-Validation. If AI can bypass the command path, it can generate documents without the required state and validation.

Impact:
`/openflow-feature` must be intercepted by command dispatch / chat hook. Formal generation, modification, and review must bind to current `state.md` / FeatureSession.

Promotion note:
Update `/openflow-feature` current workflow documentation when this feature is archived.

## Decision: Block writing-plan until feature design is complete

Scope: Candidate Global

Alternatives considered:
- Allow planning from any existing design documents.
- Require `complete` state and Cross-Validation Passed before planning.

Reason:
Planning from failed or draft-blocked feature documents would convert unresolved design gaps into implementation work.

Impact:
`/openflow-writing-plan` must reject features that are `failed`, `draft_blocked`, missing mandatory documents, or not Cross-Validation Passed after reading the document bodies.

Promotion note:
Update `/openflow-writing-plan` workflow documentation and resolver behavior.

## Decision: De-duplicate and update decisions instead of appending conflicts

Scope: Local

Alternatives considered:
- Append every new decision entry.
- Merge repeated or reversed decisions into existing entries.

Reason:
`decisions.md` should preserve durable trade-offs, not become a transcript. Duplicate or contradictory decisions make archive promotion unsafe.

Impact:
Decision writes must use title and semantic similarity to merge repeated decisions. Reversed decisions update the existing entry and preserve alternatives/reason.

Promotion note:
Not applicable.
