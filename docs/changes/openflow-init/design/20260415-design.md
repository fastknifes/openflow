# openflow/init 设计文档

- 日期：2026-04-15
- 状态：Draft
- 范围：`openflow/init` 命令、根目录 `AGENTS.md`、OpenFlow docs 导航受管区块

## 1. 背景

当前 OpenFlow 已经提供 `openflow/brainstorm`、`openflow/archive` 等面向工作流的能力，但缺少一个类似 OpenSpec 初始化入口的命令，用于在项目起始阶段为 AI 建立稳定、轻量、可持续维护的仓库级阅读约定。

目标不是把 `docs/` 内容直接塞进上下文，而是向 `AGENTS.md` 注入一段稳定的 OpenFlow 导航说明，让 AI 知道未来文档体系的语义边界，并在真实任务中按需查看相关文件。

## 2. 目标

`openflow/init` 需要满足以下目标：

- 基于 opencode 的 `/init` 命令提供包装型初始化能力。
- 为根目录 `AGENTS.md` 追加 OpenFlow 管理的 docs 导航区块。
- 明确 `docs/current`、`docs/decisions`、`docs/changes`、`docs/archive` 的职责边界。
- 告诉 AI 在不同任务下应考虑查看哪类文档，但不默认要求通读全部 docs。
- 保持命令可重复执行，且不破坏用户已有的 `AGENTS.md` 自定义内容。

## 3. 非目标

以下内容不属于本次设计范围：

- 不创建 `docs/` 目录骨架。
- 不在 init 阶段预读或预注入具体 docs 文件内容。
- 不处理子目录级 `AGENTS.md`。
- 不兼容 `agent.md`、`agents.md` 等变体文件名。
- 不将具体 OpenFlow 命令名称写入 docs 导航区块。

## 4. 核心设计

### 4.1 命令定位

`openflow/init` 是一个显式命令，不自动触发，不由系统主动提示。

它是一个自包含的命令实现：

- 首次初始化时，使用 OpenFlow 维护的静态基础模板创建 `AGENTS.md`。
- 在基础内容后追加 OpenFlow 受管区块。
- 不依赖 OpenCode 内置 `/init` 命令的实现细节。
- 如果文件写入失败，则命令失败，不做降级写入。

### 4.2 目标文件与作用范围

- 仅处理仓库根目录的 `AGENTS.md`。
- 不处理子目录级规则文件。
- 不兼容其他文件名变体。

### 4.3 首次执行与重复执行

#### 首次执行

当根目录不存在 `AGENTS.md` 时：

1. 使用 OpenFlow 静态基础模板创建 `AGENTS.md`
2. 在文件末尾追加 OpenFlow 受管区块

#### 重复执行

当根目录已存在 `AGENTS.md` 时：

- 使用 OpenFlow 静态基础模板仅在新文件创建时写入
- 已有文件首次写入后，基础内容视为用户内容，后续只刷新 OpenFlow 受管区块
- 如果文件中还没有该区块，则直接补上，不重新生成基础内容

这样可以保证初始化结果可重复执行，同时尽量不扰动已有文件主体。

### 4.4 受管区块策略

OpenFlow 注入内容采用受管区块替换模式，而不是每次追加。

约束如下：

- 使用清晰的 begin/end 标记包裹 OpenFlow 内容
- 后续执行时仅替换该区块
- `AGENTS.md` 中其他内容原样保留
- OpenFlow 区块追加在文件末尾

## 5. docs 导航内容设计

### 5.1 内容来源

docs 导航内容使用 OpenFlow 内置固定模板，不根据仓库现有 docs 结构动态生成正文。

原因：

- 该 guide 的目标是声明未来文档信息架构，而不是总结当前文件清单
- 保持输出稳定、简洁、可控
- 避免随着仓库中偶然出现的文件和目录而漂移

### 5.2 写入语言与格式

- 使用中文单语
- 使用超精简项目符号列表
- 以目录语义、查阅时机、冲突判定为核心

### 5.3 docs 信息架构

受管区块需要覆盖以下语义：

- `docs/current/requirements`：当前有效的需求事实
- `docs/current/design`：当前有效的设计事实与系统结构说明
- `docs/current/spec`：当前有效的规格说明
- `docs/current/workflow`：当前有效的流程规则与工作方式约定
- `docs/decisions/`：跨 feature 的正式全局决策
- `docs/changes/`：进行中的 feature/change 工作区，仅在处理当前变更或接手未完成工作时按需进入对应目录
- `docs/archive/`：历史归档，仅作为轻量历史参考，不作为当前实现的首选依据

### 5.4 查阅原则

受管区块需要明确写出以下原则：

- 不要默认通读全部 `docs/`
- 只有在当前任务需要时，才按需查看相关目录或文件
- 目录说明的作用是帮助 AI 决定“是否需要读”，而不是要求“先全部读一遍”

### 5.5 按任务查阅建议

受管区块需要包含任务到文档类别的轻量映射，例如：

- 涉及跨 feature 的当前需求、设计、规格与流程规则时，优先考虑查看 `docs/current/*`
- 涉及流程约定、协作规则时，优先考虑查看 `docs/current/workflow` 与 `docs/decisions/`
- 涉及当前 feature 的进行中设计或未完成工作时，按需查看对应 `docs/changes/{feature}/`
- 涉及历史追溯时，再轻量查看 `docs/archive/`

## 6. 信息冲突时的判定规则

受管区块需要明确不同文档层之间的职责边界，避免 AI 混淆来源：

- 当前系统级事实：看 `docs/current/*`
- 进行中变更：看对应 `docs/changes/{feature}/*`
- 全局规则与正式原则：看 `docs/decisions/*`
- 历史追溯：看 `docs/archive/*`

这里的规则不是简单按“哪个更新”决定，而是按任务类型决定应该参考哪一层信息。

## 7. docs 缺失时的行为

`openflow/init` 不要求仓库在执行前必须已经存在 `docs/`。

因此：

- `docs/` 不存在是正常初始状态
- `openflow/init` 不因为 `docs/` 缺失而告警
- guide 写入的是未来文档架构和按需查阅建议，而不是当前 docs 完整性校验

## 8. 命令输出

命令成功后，根据实际操作返回以下四种结果之一：

- `initialized AGENTS.md and added OpenFlow docs guide`：新创建文件并写入基础内容和受管区块
- `refreshed OpenFlow docs guide`：找到有效受管区块并替换其内容
- `appended OpenFlow docs guide`：文件无受管区块，在末尾追加新区块
- `appended fresh OpenFlow docs guide as safe repair`：检测到损坏标记，追加新区块作为安全修复

这样用户可以明确知道本次执行的操作类型。

## 9. 建议的受管区块形态

以下为建议方向，具体文案可在实现时再压缩：

```md
<!-- OPENFLOW DOCS GUIDE:BEGIN -->
- OpenFlow docs 采用按需查阅原则，不要默认通读全部 `docs/`。
- `docs/current/requirements`、`docs/current/design`、`docs/current/spec`、`docs/current/workflow` 用于当前有效事实与当前流程约定。
- `docs/decisions/` 用于跨 feature 的正式全局决策。
- `docs/changes/` 用于进行中的 feature/change 工作区，仅在处理当前变更或接手未完成工作时按需查看对应目录。
- `docs/archive/` 仅用于历史追溯，不作为当前实现的首选依据。
- 当信息看起来不一致时，按任务类型判断来源：当前事实看 `docs/current/*`，进行中变更看对应 `docs/changes/{feature}/*`，全局规则看 `docs/decisions/*`，历史追溯才看 `docs/archive/*`。
<!-- OPENFLOW DOCS GUIDE:END -->
```

## 10. 验收重点

后续实现时，至少需要验证以下场景：

1. 根目录不存在 `AGENTS.md` 时，使用 OpenFlow 基础模板创建文件并追加受管区块。
2. 根目录已存在 `AGENTS.md` 且已含有效 OpenFlow 区块时，仅刷新该区块，保留其他内容。
3. 根目录已存在 `AGENTS.md` 但没有 OpenFlow 区块时，仅补充该区块。
4. 标记损坏（只有开始标记、只有结束标记、重复区块、嵌套等）时，在文件末尾追加新区块作为安全修复，不就地重写损坏区域。
5. 不存在 `docs/` 时不报错、不告警，仍可成功完成 guide 写入。
6. 重复执行时不破坏 `AGENTS.md` 其他内容。
7. 保留现有文件的换行符风格（`\r\n` 或 `\n`）。

## 11. 结论

本设计将 `openflow/init` 定义为一个轻量、稳定、幂等的仓库级初始化增强命令。它不负责灌输大量文档内容，而是负责把未来的文档信息架构和查阅原则准确地放进 `AGENTS.md`，从而让 AI 在后续工作流中按需读取、按任务判断、避免上下文失控。
