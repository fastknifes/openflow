## Design Context

> Auto-injected from the active OpenFlow design workspace for `openflow-issue`.
> Refer to the original design documents for full details.

#### Design
### design.md

### 1. 问题陈述
OpenFlow 当前主流程更适合 feature/change：

```text
brainstorm → implement → harden → verify → archive
```

该流程的核心价值是通过文档约束 AI，让实现符合设计、需求、当前事实和全局决策。但用户在实际开发中经常输入的是“不确定性质的问题”，例如：

- 某个接口结果不对
- 某条线上数据展示异常
- 某个状态没有流转
- 某个用户看不到预期信息
- 某段历史数据和当前规则冲突

这些输入不能直接假设为 bug。它们可能是：

- 实现缺陷
- 数据问题
- 配置问题
- 环境问题
- 文档语义不清
- 用户期望变化
- 当前业务规则需要重新裁决

如果 AI 在未澄清需求、约束和当前项目语义前直接修代码，容易出现“修掉现象，但破坏语义”的问题。典型风险包括：

1. 将数据异常误判为代码缺陷
2. 将需求变更伪装成 bugfix
3. 在没有业务裁决的情况下改变用户可见行为
4. 修复单一路径但遗漏同语义下的其他入口
5. 修复后无法说明它是否符合 `docs/current/*` 和 `docs/decisions/*`

因此需要一个面向“不确定问题”的单命令入口，用于需求澄清、约束澄清、证据调查、语义对齐和问题分诊。

---

### 2. 设计目标
新增命令：

```text
/openflow-issue <issue-name-or-description>
```

该命令不直接等同于 bug 修复命令，也不替代 `/openflow-brainstorm`。它的目标是：

> 接收一个不确定性质的问题，先明确用户期望、修复边界、当前项目语义和证据，再判断下一步是直接修复、处理数据、修配置、继续调查、询问用户，还是升级到 brainstorm。

### 2.1 核心目标
1. **不预设问题类型**
   - 用户输入 issue 时，系统不能默认它是 bug。

2. **需求澄清前置**
   - 先明确用户认为哪里不对、期望行为是什么、该期望是否属于已有系统语义。

3. **约束澄清前置**
   - 明确哪些数据、接口、历史行为、用户可见行为和业务规则不能被改变。

4. **证据调查只读化**
   - 调查阶段只能读取代码、文档、日志、配置、测试和数据；不能修改代码或数据。

5. **语义对齐**
   - 修复前必须判断当前项目语义是什么，以及现象是否违反该语义。

6. **分类分诊**
   - 将问题分类为 bugfix、data_issue、config_issue、environment_issue、doc_ambiguity、behavior_change 或 cannot_determine。

7. **明确下一步 gate**
   - 输出允许动作、禁止动作、是否需要用户澄清、是否需要 brainstorm、是否建议 harden/verify/archive。

8. **语义治理提升**
   - 当 issue 暴露出需求未澄清、约束不足或当前语义空白时，澄清结果不能只服务本次修复，必须形成可提升到 `docs/current/*` 或 `docs/decisions/*` 的治理候选。

9. **保持用户入口简单**
   - 对用户只暴露一个命令，内部多阶段执行。

---

### 3. 非目标
`/openflow-issue` 不做以下事情：

1. 不替代 `/openflow-brainstorm`
2. 不直接实现修复
3. 不默认生成完整 feature design
4. 不默认 archive
5. 不作为第二套 OpenFlow orchestrator
6. 不在调查阶段修改代码、数据或配置
7. 不把所有 issue 都强制升级为 feature/change
8. 不对业务语义自行拍板
9. 不把未经用户确认的业务规则直接写入 `docs/decisions/*`

---

### 6.4 约束
- 不能假设问题是 bug
- 不能假设必须改代码
- 不能在 intake 阶段给出修复结论

## Context

### Original Request
User requested a development plan for implementing the design in `docs/changes/2026-05-09-openflow-issue/design.md`.

### Interview Summary
- The command should be named `/openflow-issue`, not `/openflow-bug`, because the user cannot know whether a submitted problem is truly a bug.
- The command should be a single user-facing entry point with internal phases, not a separate heavy workflow.
- The command should focus on requirement clarification, constraint clarification, evidence collection, semantic alignment, classification, and governance promotion.
- If an issue exposes unclear requirements or missing constraints, clarified semantics should become current facts or confirmed global decisions, not remain one-off repair notes.

### Metis Review (gaps addressed)
- **Identity/state model must be explicit**: this plan chooses a safe issue slug plus date-prefixed workspace; raw case text is preserved in artifacts.
- **Classification must be explicit**: this plan chooses multi-label classification with a required primary classification for gating.
- **Read-only guardrails must be executable**: this plan includes readonly/prod rules and tests blocking writes during investigation.
- **Verify/archive must be re-scoped before final command completion**: this plan creates issue-aware mode without breaking feature mode.
- **Governance promotion must not auto-activate decisions**: this plan requires explicit user approval before writing `docs/decisions/*`.

## Work Objectives

### Core Objective
Implement `/openflow-issue` so OpenFlow can safely process uncertain problems before code changes, classify their nature, produce issue artifacts, and route them to direct fix, data/config/environment handling, user clarification, brainstorm, verify, archive, or governance promotion.

- Tests under `tests/commands/`, `tests/hooks/`, `tests/utils/`, and registration/runtime test coverage.

### Must Have
- One command: `/openflow-issue <case>`.
- Eight phases reflected in output/artifacts: Issue Intake, Requirement Clarification, Constraint Clarification, Evidence Investigation, Semantic Alignment, Classification, Next Action Gate, Governance Promotion.
- Multi-label classification support with exactly one primary classification for gating.
- `bugfix` is allowed only when semantics are clear and the action restores existing behavior.
- `doc_ambiguity` and `behavior_change` block implementation/archive until user clarification, brainstorm, or decision path is chosen.
- Production and `--readonly` modes never modify code/data/config.
- `docs/decisions/*` updates require explicit user approval.
- Existing feature brainstorm/verify/archive behavior remains compatible.

### Must NOT Have
- No direct code fix execution inside `/openflow-issue`.
- No destructive shell/database/data operations during investigation.
- No automatic global decision creation without user confirmation.
- No forced brainstorm for all issues.
- No regression to existing slash command registration or feature-mode verify/archive tests.
